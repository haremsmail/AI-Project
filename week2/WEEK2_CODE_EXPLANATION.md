## 2) File-by-File Meaning

### main.py
Purpose:
- Creates the GUI window, tabs, buttons, status labels, and board drawing.
- Starts solving in a background thread so UI does not freeze.
- Shows solution steps with Next Step and Previous Step.
- Saves the final output to solution.txt.

Important flow:
1. App starts, creates two tabs:
   - 8-Puzzle Solver
   - Vacuum World Solver
2. Generate buttons create random initial states.
3. Solve button:
   - Reads selected algorithm.
   - Runs solver in a background thread.
   - On finish, updates GUI on main thread.
4. If solved:
   - Enables step navigation.
   - Writes solution.txt automatically.
5. If vacuum has no solution:
   - Shows no-solution status.
   - Writes required no-solution text to solution.txt.

### puzzle_solver.py
Purpose:
- Defines PuzzleState (board, neighbors, goal check, Manhattan heuristic).
- Defines PuzzleSolver wrapper methods for DFS, BFS, Best-FS.
- Builds a random solvable 8-puzzle each run.
- Saves puzzle report to solution.txt.

Key points:
1. Goal board is fixed as:
   1 2 3
   4 5 6
   7 8 0
2. Neighbors are generated by sliding blank tile up/down/left/right.
3. Every puzzle move has cost 1.
4. Random board generation is done by shuffling from goal using legal moves, so puzzle remains solvable.

### vacuum_solver.py
Purpose:
- Defines VacuumState (grid, vacuum position, movement, goal check).
- Defines VacuumSolver wrapper methods for DFS, BFS, Best-FS.
- Generates random world with obstacles, dirt, and vacuum start.
- Saves vacuum report to solution.txt.

Key points:
1. Goal is when vacuum position reaches dirt position.
2. Obstacles cannot be crossed.
3. Movement costs are implemented exactly as required:
   - Up: 2
   - Down: 0
   - Left: 1
   - Right: 1
4. If no path exists because of obstacles:
   - has_solution becomes false.
   - solution.txt contains the no-solution explanation.

### search_algorithms.py
Purpose:
- Shared implementation of DFS, BFS, and Best-FS.
- Used by both puzzle and vacuum solvers.

Algorithm behavior:
1. DFS:
   - Uses stack (LIFO).
   - Goes deep before backtracking.
   - Can return very long solutions.
2. BFS:
   - Uses queue (FIFO).
   - Expands by depth level.
   - Finds shortest path by number of steps.
3. Best-FS:
   - Uses priority queue by heuristic value.
   - Tries promising states first.

Implementation detail:
- Stores parent links and reconstructs final path when goal is found.
- This is more memory-efficient than copying full path each expansion.

## 3) What Happens When You Click Solve

For 8-puzzle tab:
1. GUI checks no other solve is running.
2. GUI disables Solve and Generate buttons.
3. Status changes to Solving.
4. Background thread calls one of:
   - solve_dfs
   - solve_bfs
   - solve_best_fs
5. If success:
   - GUI receives all states in solution path.
   - Draws first state in solution view.
   - Enables Next Step and Previous Step.
   - Saves solution.txt.
6. If error/no solution:
   - GUI shows message and re-enables controls.

For vacuum tab:
1. Same thread logic.
2. If path exists, steps are displayed and saved.
3. If path does not exist, no-solution message is shown and saved.

## 4) Why DFS Steps Can Be Very Long

DFS is correct but not optimal.
- It finds a valid path, not necessarily the shortest path.
- On some random 8-puzzle states, DFS may produce thousands of steps.

This does not mean code is broken.
It means DFS nature is deep exploration.

If you want shorter demonstrations, use BFS or Best-FS.
